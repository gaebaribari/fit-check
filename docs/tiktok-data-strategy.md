# TikTok 데이터 수집 전략 검토

## 현재 방식: 직접 스크래핑

### 구조
```
Next.js 서버 → TikTok HTML 직접 요청 → __UNIVERSAL_DATA_FOR_REHYDRATION__ 파싱 (프로필)
Next.js 서버 → yt-dlp 실행 → 최근 영상 썸네일/게시일 조회
```

### 장점
- 무료
- 외부 의존성 없음 (yt-dlp 제외)

### 단점
- **TikTok 봇 차단**: 동일 IP에서 반복 요청 시 차단됨 (일시적 또는 영구적)
- **불안정**: HTML 구조 변경 시 파싱 깨짐
- **속도 제한**: 차단 회피를 위해 요청 간 딜레이 필요 → 대량 조회 시 느림

### 현재 대응
- 실패 시 최대 2회 재시도 (1~3초 대기)
- User-Agent 5종 랜덤 로테이션
- 3명씩 병렬 처리

### 리스크
- 조회량이 늘어나면 차단 빈도 증가
- TikTok HTML 구조 변경 시 즉각 대응 필요

## 대안 1: 서드파티 API (권장)

### 구조
```
Next.js 서버 → 서드파티 API 호출 → JSON 응답 (프로필 + 영상 정보)
```

### 원리
- 서드파티 업체가 수백~수천 개의 프록시 IP를 활용하여 TikTok 데이터를 수집
- 우리는 해당 업체의 API를 호출하고 깔끔한 JSON 데이터를 받음
- 차단 관리, IP 로테이션 등은 업체가 담당

### 장점
- **안정성**: 차단 걱정 없음, 업체가 관리
- **속도**: 최적화된 인프라, 빠른 응답
- **유지보수 불필요**: TikTok 구조 변경 시 업체가 대응
- **yt-dlp 의존성 제거**: 서버에 yt-dlp 설치 불필요

### 단점
- **비용**: 월 $10~50+ (사용량에 따라)
- **업체 의존**: 업체 서비스 중단 시 대체 필요

### 대표 서비스
- RapidAPI TikTok 관련 API들
- 기타 TikTok 데이터 전문 업체

### 연동 난이도
- **낮음**: 현재 코드에서 `fetchProfileInfo()` 함수 하나만 교체하면 됨
- 기존 SSE 스트리밍, 필터링, CSV 내보내기 등은 그대로 유지

## 대안 2: TikTok 공식 API

### 종류
| API | 대상 | 승인 난이도 |
|---|---|---|
| Research API | 학술/연구 기관 | 높음 (학술 소속 필요) |
| Commercial Content API | 비즈니스 파트너 | 높음 (파트너 승인) |
| Marketing API | 광고주 | 중간 (비즈니스 계정 필요) |

### 장점
- 공식 지원, 가장 안정적
- 차단 걱정 없음

### 단점
- 승인 과정이 길고 까다로움
- 제공 데이터 범위가 제한적일 수 있음

---

## 대안 3: 자체 프록시 구축

### 구조
```
Next.js 서버 → 프록시 풀 (수백 개 IP) → TikTok
```

### 장점
- 외부 업체 의존 없음
- 완전한 제어

### 단점
- **프록시 비용**: 월 $20~100+ (서드파티 API와 비슷하거나 더 비쌈)
- **관리 부담**: IP 차단 모니터링, 교체, 로테이션 로직 직접 구현
- **개발 리소스**: 서드파티 API 대비 투자 대비 효과 낮음
